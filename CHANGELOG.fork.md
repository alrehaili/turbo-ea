# Changelog (Fork)

Changes specific to this fork. For upstream changes, see [CHANGELOG.md](CHANGELOG.md).

## [Unreleased]

### Added
- **NORA demo dataset (`SEED_NORA=true`).** One environment flag seeds a fictional Saudi agency landscape that populates every NORA view: four Government Services wired to processes, applications, capabilities and data; NDMO-classified Data Objects; two Data Exchanges (one secret exchange off-GSB that lights up the Interoperability warning); a current→target replacement pair plus a planned retirement (populating Gap Analysis with traceable and untraceable changes); three KPIs landing green/amber/red on the scorecard; Stage 1–3 progress with evidence links on the NORA Program tracker; an in-transition improvement opportunity; and a draft EA Project Strategy document. Applies the NORA profile automatically, is idempotent, and is machine-checked against the metamodel by a dedicated compatibility test suite.
- **Data Exchange and KPI card types (NORA DRM + PRM).** The NORA profile now creates two more types: **Data Exchange** (method, frequency, external party, GSB routing, NDMO classification carried) wired to Applications (with a sends/receives/bidirectional direction attribute) and Data Objects; and **KPI** (unit, baseline/target/current values, measurement frequency, direction) wired to Objectives, Government Services and Initiatives. A **Database** subtype lands on IT Component and Data Objects gain a "is stored in" relation to IT Components — completing the DRM database-portfolio artifact. All translated ×10.
- **KPI Scorecard and Interoperability reports.** **Reports → KPI Scorecard** shows every KPI with direction-aware progress and RAG colouring; **Reports → Interoperability** lists every interface and data exchange with classification, GSB routing, external parties and connected applications — and flags secret-or-above exchanges running outside the Government Service Bus.
- **Saudi compliance regulation pack.** Applying the NORA profile seeds four scanner regulations: **NCA Essential Cybersecurity Controls**, **NDMO Data Management standards**, **PDPL** and **DGA Digital Government Policy checks** — their assessment scopes key on the NORA profile fields (NDMO classification, GSB flags, BRM/ARM linkage, standards status), so a compliance scan can flag e.g. secret data exchanged off-GSB or services without a digital channel.
- **Organization Chart and Service Traceability reports.** Two new reports complete the NORA artifact views: **Reports → Organization Chart** renders the Organization hierarchy as an indented tree (the NORA BA org-chart artifact), and **Reports → Service Traceability** answers "how is this government service delivered" — pick a service and see every process, application, data object and technology within two relation hops, grouped into the four EA layers (target-state cards render dashed). Deep-linkable via `?card_id=…`.
- **Committee decision register on ADRs.** Architecture decisions now carry the deciding committee, meeting date and NORA stage — so ADRs double as the EA Governance Committee's decision log. Editable in the ADR editor, preserved across revisions.
- **NORA governed documents (EA Strategy / Plan / SWOT / Usage / Management plans).** The SoAW machinery — rich-text sections, revision chain, signatories, DOCX/PDF export — now also authors the five NORA Stage 1/2/3/9 documents, each with its own section template (e.g. Environment Analysis carries EA Requirements, Internal/External Environment and the four SWOT quadrants). Create them from the **NORA Program → New NORA document** menu; the document type is shown in delivery lists and filterable via the API (`GET /soaw?doc_type=…`). Signed documents revise into new versions exactly like SoAWs. Templates translated in all 10 locales.
- **NORA EA Program tracker.** New **NORA Program** page (visible when the NORA profile is active) tracking all ten methodology stages plus continuous governance: 41 seeded deliverables straight from the NORA guideline, per-stage progress bars, status workflow (not started → in progress → in review → approved, or descoped for tailoring), evidence links on every deliverable (app paths or URLs), custom deliverables per stage, and stage-gate approval — marking a deliverable approved requires the `governance.approve_step` permission and stamps who approved it when. New `nora.view` / `nora.manage` permissions; deliverables travel in the workspace-transfer bundle.
- **Improvement Opportunity registry.** New **GRC → Governance → Opportunities** tab (NORA Stage 6.6): governable improvement records classified by architecture domain (BA/AA/DA/TA) with priority and lifecycle (proposed → approved → in transition → realized / rejected), linked cards, and one-click assignment of the realising transition initiative (which automatically moves the opportunity to "in transition").
- **Architecture state on cards (current / transition / target).** Every card now carries an architecture-state slice plus a typed change marker (create / modify / replace / retire / consolidate) and an optional successor link — so a 3–5-year target blueprint lives side-by-side with the live landscape on the same inventory (NORA Stages 6–7, TOGAF baseline/target). A state badge on the card detail header switches slices; the inventory API filters by state; promoting a target card to current emits a dedicated audit event; state and change type travel in the workspace-transfer bundle.
- **Gap Analysis report.** New report under **Reports → Gap Analysis** bucketing every current→target delta (new / replacements / modifications / retirements), showing which transition initiative delivers each change (with its transition role), flagging **untraceable** changes no initiative owns, and offering one-click "Assign to initiative". Initiative relation types gain a `transitionRole` attribute (introduces / modifies / retires) when the NORA profile is applied.
- **Multi-step governance approval (stage gates).** Optional governance mode (**Admin → Settings → Governance Workflow**) replaces one-click card approval with an ordered review chain — by default Chief Architect, then EA Governance Committee. Cards are *submitted* for review (new IN_REVIEW status), each chain step is decided by users holding that role plus the new `governance.approve_step` permission, segregation of duties blocks submitters from approving their own round, mid-review edits restart the round, chain members get in-app notifications, and the card detail shows the live review-chain stepper. The classic flow is untouched when the mode is off.
- **NORA governance role pack.** Applying the NORA profile seeds three roles: **EA Working Team** (drafts, cannot approve), **Chief Architect** (authoring + first review step) and **EA Governance Committee** (read access + final approval).
- **NORA TRM metadata on Technology Standards.** The Technology Standards catalogue now carries the fields NORA's Technology Reference Model expects on a Service Standard: issuing body (DGA, NCA, W3C, …), mandate level (mandatory / recommended / optional), next review date, specification URL, TRM code, and an optional link to a **Tech Category** card so standards slot into the TRM Service Area / Category tree. All fields are editable in the standards dialog (with a Tech Category picker) and returned by the list / radar / detail endpoints; the Tech Category link participates in workspace export/import. Translated in all 10 locales.
- **Government Service card type (NORA Service Catalogue).** Applying the NORA profile now also creates a `GovService` card type in the Business Architecture layer — service code, beneficiary types (citizen / resident / business / government / visitor), delivery channels (portal / mobile app / service center / call center / kiosk), service maturity (informational → proactive), fee model, SLA and volume metrics — plus five relation types wiring services to Business Processes ("is realized by"), Applications ("is delivered by"), Organizations ("provides"), Business Capabilities ("supports") and Data Objects ("uses"), and a `service_owner` stakeholder role. Relation creation respects the one-relation-type-per-pair rule: pairs an admin already modelled are skipped, never duplicated. Fully translated in all 10 locales.
- **NORA framework profile (Saudi National Overall Reference Architecture).** A new opt-in framework profile under **Admin → Settings → Modules → Framework Profile** (or `SEED_PROFILE=nora` for fresh installs) injects the NORA/NEA alignment fields into the built-in card types as a "NORA Alignment" section: BRM code/level on Business Capabilities, ARM code/category, automation level and national-shared-service flag on Applications, TRM code/hosting model/security zone on IT Components, DRM code, NDMO data classification (Top Secret / Secret / Restricted / Public), PII flag, authoritative-source flag and retention period on Data Objects, integration type/authentication and Government Service Bus (GSB) flag on Interfaces, and national-strategy alignment on Objectives. Fully translated in all 10 locales (Arabic first-class), idempotent, additive-only (weight 0 — data-quality scores are unaffected), and reversible without data loss. See `noraPlan.md` (WP1.1) for the full NORA roadmap.
- **Architecture Standards, linked to Principles.** A new managed entity captures the concrete standards that implement your architecture principles (e.g. "Approved RDBMS is PostgreSQL"). Manage them under **Admin → Metamodel → Standards** — each standard can be linked to one or more EA principles — and browse the active set under **GRC → Governance → Standards**. Standards (and their principle links) are included in the workspace export/import bundle.
- **Manage principles and standards inline from GRC → Governance.** Users with the *Metamodel* admin permission now get the full create / edit / activate / delete controls directly on the **GRC → Governance → Principles** and **Standards** sub-tabs — the same management UI as Admin → Metamodel — so governance authors no longer have to switch pages. Users without that permission continue to see the read-only list.
