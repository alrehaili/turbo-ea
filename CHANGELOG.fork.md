# Changelog (Fork)

Changes specific to this fork. For upstream changes, see [CHANGELOG.md](CHANGELOG.md).

## [2.22.0] - 2026-07-20

### Added
- **Reference Models — governance & versioning (RMPlan Phase 5, §9 + §15).** Publishing a reference model now records a preserved **version snapshot** of its item tree (new `reference_model_versions` table, migration 157), so a published model is never silently overwritten. A new **Version history** dialog on the management console lists every published version (summary, publisher, item count) and **compares** any snapshot against the live model or another snapshot — showing added / removed / changed items by code. A lightweight **review workflow** is added between draft and publish: a new `in_review` status with **Submit for review** (`reference_models.manage`) and **Reject → back to draft** (`governance.approve_step`); publishing still requires `governance.approve_step` and now accepts an optional change summary. New endpoints (`POST …/submit`, `POST …/reject`, `GET …/versions`, `GET …/versions/{id}`, `GET …/versions/{id}/diff`); versions transfer with the workspace. Translated EN + AR.
- **Reference Models — coverage & gap analysis (RMPlan Phase 4, §8).** The browse page gains a **Coverage** view (fifth toggle) that turns a model into decision support: KPI tiles (coverage %, covered / uncovered / duplicate-support / retiring-only capabilities, unmapped inventory), a **gap list** grouped by kind, and a per-capability **coverage matrix**. Gaps are computed server-side by `GET /reference-models/{id}/gaps` (`reference_models.view`): a leaf with no mapping (uncovered), a leaf with ≥2 mapped cards (possible duplication), and a leaf whose every mapped card is phasing out / end-of-life (lifecycle risk). Gap and matrix rows open the item side panel. Translated EN + AR.
- **Reference Models — banded seeds + demo links across all six domains.** DRM, TRM, BXRM and SRM starter models are restructured into the same banded **band → category → capability** shape as BRM (bilingual EN + AR — BXRM/SRM previously had no Arabic), so every domain renders as a proper capability-map poster. `SEED_NORA` now seeds **demo inventory cards for every domain** (Application/DataObject/ITComponent/GovService/SecurityControl, guarded on the card type existing) mapped via the code fields, with one retiring card per domain to exercise gap analysis — so all six maps show real coverage out of the box (208 reference-model items, 45 demo cards total). (Existing installs keep their seeded models — re-import to adopt the banded structure.)
- **Reference Models — banded dark capability-map (RMPlan Phase 3, §18).** The Map view now renders a model's hierarchy as the dark teal capability-map poster: top-level items become full-width **bands** with a dark header + coverage count, each band's children become titled **category** columns, and each category's children become clickable **capability boxes** (icon + name + mapped-inventory count, colored coverage accent). Two-level models degrade to a band of boxes; a **bottom domain bar** switches between the six reference-model domains. Fully generated from stored data, RTL-aware. The **BRM starter seed** is restructured into this banded band → category → capability shape (4 bands — Administrative & Regulatory, Core, Operational, Enabling — 107 bilingual items), and `SEED_NORA` now also seeds 13 demo **BusinessCapability** cards mapped to BRM leaves (via `brmCode`) so the map shows real coverage out of the box. (Existing installs keep their seeded models — re-import to adopt the banded structure.)
- **Reference Models — full "poster" view with editable narrative (RMPlan Phase 3, §18).** The **Map** view is now a complete reference-model poster: the colored capability map is wrapped by editable **narrative panels** (mission, vision, mandate, strategic objectives, key stakeholders, guiding principles, KPIs, value/outcomes — or any custom panels), matching the reference-image slides but generated entirely from stored, bilingual data rather than a static image. Panels are authored in the management console via a new **Poster content** editor (add / remove / reorder, text or bulleted-list panels, EN + AR, placement in the header band or the grid below the map, plus a one-click **starter set**). Header-placement panels render above the map; grid panels below. Backed by a new `narrative` JSONB column on `reference_models` (migration 156) and `PATCH /reference-models/{id}/narrative` (`reference_models.manage`); the column transfers with the workspace automatically. Translated EN + AR, RTL-aware.
- **Reference Models — visual capability-map view (RMPlan Phase 3, §5.A).** Underlying the poster, the browse page gained a **Map** view mode (fourth toggle, alongside Tree / Table / Unmapped) rendering the model's hierarchy as a colored capability map — top level as colored columns, children as boxes (deeper descendants as compact lines), each showing its mapped-inventory count and a coverage dot (green = directly mapped, amber = covered via children, grey = none). Single-root models fan their root's children out into columns. Every box is clickable, opening the same item side panel (details + mapped cards + Map-a-card) the other views use. Column palette lives in the design tokens; RTL-aware.
- **Reference Models — inventory mapping (RMPlan Phase 2).** Reference-model items can now carry **explicit mappings** to inventory cards beyond the single primary classification the card-detail code field expresses: a new `reference_model_mappings` table (migration 155) records an M:N card↔item link with a **mapping type** (primary / secondary / supporting / candidate / historical), a **status** (proposed / confirmed / rejected), a free-text **rationale** and an optional 0–100 **confidence**. On the browse page an item's side panel gains **Map a card** (built on the shared `CardPicker`, scoped to the domain's card type) plus per-mapping edit/remove, mapping-type chips, and a new **Unmapped** view listing cards of the domain type that resolve to no component. Coverage counts everywhere (landing, overview, item tree) now **merge** code-field classification with explicit mappings, deduped by card. New endpoints under `reference_models.view` (`GET …/{id}/unmapped-inventory`, `GET …/items/{item_id}/mappings`) and a new `reference_models.map` permission (granted to member + bpm_admin) gating the mapping mutations (`POST …/items/{item_id}/mappings`, `PATCH`/`DELETE …/mappings/{id}`). Every mutation emits an audited `reference_model_mapping.*` event stamped on the linked card's history. Mappings transfer with the workspace (card FK remapped, user FKs email-remapped). Translated EN + AR.
- **Reference Models browse experience (RMPlan Phase 1).** A new top-level **Reference Models** navigation dropdown (NORA profile only) opens a landing page with one card per NEA domain — published model, version, status, item count, mapped-inventory count, coverage % — and per-domain browse pages at `/reference-models/{domain}` showing the published model as an expandable **tree** or a flat **table** (view mode, search and selection persist in the URL). Every item carries mapped-inventory counts (direct + descendant-inclusive) computed from the card code attributes (`brmCode`/`armCode`/`drmCode`/`trmCode`/`bxrmCode`/`srmCode`); selecting an item opens a side panel with its details and the actual inventory cards mapped to it, each linking into card detail. Backed by three new read endpoints (`GET /reference-models/overview`, `GET /reference-models/{id}/summary`, `GET /reference-models/{id}/items/{item_id}/cards`), all gated by `reference_models.view`. The management console moves to `/reference-models/manage` (Governance menu entry now `reference_models.manage`-gated). Translated EN + AR, RTL-aware (codes stay LTR).

### Fixed
- **Promoting a target card to current crashed.** The `POST /cards/{id}/promote-target` endpoint referenced an unimported `UUID` name and a non-existent response helper, so every call failed with an internal error. It now parses the id correctly and returns the standard card response.
- **The app failed to compile after the segment-filter work.** The Phase B.9 frontend landed with twenty TypeScript errors (missing `segmentIds` in the inventory filter initialisers, untyped bootstrap responses, unused imports, a dead menu referencing a non-existent grid property, and the per-type-approval-chains dialog referencing metamodel types that were never loaded) — `npm run build` was broken at branch head. All fixed; the governance settings dialog now actually lists the card types.
- **Missing Arabic UI strings.** 23 Arabic keys that had shipped English-only (architecture state / change type labels, inventory columns and filters, the per-type approval-chain settings, an approval-step notification preference, and the common Show label) are now translated, restoring the i18n parity suite to green.

- **Assigning the EA Governance Committee role to a user failed.** `users.role` was a 20-character column while the NORA governance pack seeds the 23-character `ea_governance_committee` role key — every attempt to give a user that role failed with a database truncation error. The column is widened to 50 characters (matching role keys themselves), with a migration (136) for existing installs.
- **Gap Analysis, Service Traceability and Interoperability reports were admin-only by mistake.** The three NORA report endpoints checked a permission key that doesn't exist (`reports.view`), so every non-admin got a 403. They now check `reports.ea_dashboard` like every other report.
- **Filtering the inventory by architecture state could crash, and approval-status filters returned wrong totals.** The card-list endpoint's count query applied the approval-status filter in the architecture-state branch (crashing with an internal error when filtering by state alone) and never applied it in the approval-status branch (wrong pagination totals). Both filters now apply to both queries.

Crucially, suggestions are **advisory only**: they are shown for human review with checkboxes, and only the ones you accept are created — landing as `proposed` records (source `ai`) so governance approval stays a human decision. New `POST /improvement-opportunities/ai-suggest` endpoint gated by `grc.manage` + `ai.suggest`. Implements noraPlan.md WP5.5.
- **NORA plateaus (time-slices) & segment scopes.** A new **Plateaus & Segments** section on the NORA Program page adds two governance overlays on the single canonical landscape. **Segments** are card-rooted, reusable filter scopes: pick a root card (typically a Business Capability), optionally pull in its hierarchy descendants and related cards (optionally narrowed to specific card types), and view the resolved scope grouped into the four EA layers with clickable card chips — the blueprint's *ArchitectureScope* slimmed to a filter-set entity. **Plateaus** are named target dates (e.g. "2027 Interim", "2030 Target"); the time-slice view reclassifies every card's lifecycle phase (plan → phase-in → active → phase-out → end-of-life) *as of* that date and breaks the landscape down by phase and architecture state — a point-in-time view without duplicating a single card. Both are full CRUD, gated by the existing `nora.view` / `nora.manage` permissions, and travel in the workspace-transfer bundle (segment root-card FKs remapped on import). Implements noraPlan.md WP5.4.
- **NEA alignment / evidence-pack export.** A new **NEA Evidence Packs** section on the NORA Program page generates an immutable, multi-sheet `.xlsx` snapshot of the agency's NORA posture for NEA federation and audit: an **Overview** of headline metrics, **EA Maturity** (latest assessment, per-dimension level vs target), **Program Status** (deliverables by stage), **BRM Coverage** (capabilities mapped to BRM codes + coverage %), **Shared Services** (GovServices/Applications flagged shared), **Standards Compliance** (standards, mandate, open exceptions) and **Approval History** (recent governance events). Packs are generated on demand, listed with headline chips (maturity / BRM / program %), downloadable, and deletable — but never editable once produced. The workbook binary is stored on disk (like workspace transfers) with a `nea_evidence_packs` row tracking the lifecycle; each sheet degrades gracefully so a pack can be produced at any stage of a deployment. Gated by the existing `nora.view` (list/download) and `nora.manage` (generate/delete) permissions. Implements noraPlan.md WP5.3.
- **EA maturity self-assessment (Qiyas-style).** New **EA Maturity** page (visible when the NORA profile is active) for scoring the EA function across admin-definable maturity dimensions on a fixed 1–5 scale. Ships ten seeded NORA/Qiyas dimensions (governance, the four architecture domains, security & compliance, methodology, performance, change & transition, national alignment) that admins can extend or deactivate. Create dated assessments, score each dimension's current level and target, and the page renders a **radar** (latest assessment, level vs target) plus a **trend line** of the weighted overall score across assessments. Assessments move draft → submitted → approved (approval requires `governance.approve_step`); a dimension gap can be promoted straight into the Improvement Opportunity registry with one click. New `maturity.view` / `maturity.manage` permissions (granted to the NORA working-team and chief-architect roles); dimensions, assessments and scores all travel in the workspace-transfer bundle. Implements noraPlan.md WP5.2.
- **NORA demo dataset (`SEED_NORA=true`).** One environment flag seeds a fictional Saudi agency landscape that populates every NORA view: four Government Services wired to processes, applications, capabilities and data; NDMO-classified Data Objects; two Data Exchanges (one secret exchange off-GSB that lights up the Interoperability warning); a current→target replacement pair plus a planned retirement (populating Gap Analysis with traceable and untraceable changes); three KPIs landing green/amber/red on the scorecard; Stage 1–3 progress with evidence links on the NORA Program tracker; an in-transition improvement opportunity; and a draft EA Project Strategy document. Applies the NORA profile automatically, is idempotent, and is machine-checked against the metamodel by a dedicated compatibility test suite.
- **Data Exchange and KPI card types (NORA DRM + PRM).** The NORA profile now creates two more types: **Data Exchange** (method, frequency, external party, GSB routing, NDMO classification carried) wired to Applications (with a sends/receives/bidirectional direction attribute) and Data Objects; and **KPI** (unit, baseline/target/current values, measurement frequency, direction) wired to Objectives, Government Services and Initiatives. A **Database** subtype lands on IT Component and Data Objects gain a "is stored in" relation to IT Components — completing the DRM database-portfolio artifact. All translated ×10.
- **KPI Scorecard and Interoperability reports.** **Reports → KPI Scorecard** shows every KPI with direction-aware progress and RAG colouring; **Reports → Interoperability** lists every interface and data exchange with classification, GSB routing, external parties and connected applications — and flags secret-or-above exchanges running outside the Government Service Bus.
- **Saudi compliance regulation pack.** Applying the NORA profile seeds four scanner regulations: **NCA Essential Cybersecurity Controls**, **NDMO Data Management standards**, **PDPL** and **DGA Digital Government Policy checks** — their assessment scopes key on the NORA profile fields (NDMO classification, GSB flags, BRM/ARM linkage, standards status), so a compliance scan can flag e.g. secret data exchanged off-GSB or services without a digital channel.
- **Organization Chart and Service Traceability reports.** Two new reports complete the NORA artifact views: **Reports → Organization Chart** renders the Organization hierarchy as an indented tree (the NORA BA org-chart artifact), and **Reports → Service Traceability** answers "how is this government service delivered" — pick a service and see every process, application, data object and technology within two relation hops, grouped into the four EA layers (target-state cards render dashed). Deep-linkable via `?card_id=…`.
- **Committee decision register on ADRs.** Architecture decisions now carry the deciding committee, meeting date and NORA stage — so ADRs double as the EA Governance Committee's decision log. Editable in the ADR editor, preserved across revisions.
- **NORA governed documents (EA Strategy / Plan / SWOT / Usage / Management plans).** The SoAW machinery — rich-text sections, revision chain, signatories, DOCX/PDF export — now also authors the five NORA Stage 1/2/3/9 documents, each with its own section template (e.g. Environment Analysis carries EA Requirements, Internal/External Environment and the four SWOT quadrants). Create them from the **NORA Program → New NORA document** menu; the document type is shown in delivery lists and filterable via the API (`GET /soaw?doc_type=…`). Signed documents revise into new versions exactly like SoAWs. Templates translated in all 10 locales.
- **NORA EA Program tracker.** New **NORA Program** page (visible when the NORA profile is active) tracking all ten methodology stages plus continuous governance: 41 seeded deliverables straight from the NORA guideline, per-stage progress bars, status workflow (not started → in progress → in review → approved, or descoped for tailoring), evidence links on every deliverable (app paths or URLs), custom deliverables per stage, and stage-gate approval — marking a deliverable approved requires the `governance.approve_step` permission and stamps who approved it when. New `nora.view` / `nora.manage` permissions; deliverables travel in the workspace-transfer bundle.
- **Improvement Opportunity registry.** New **GRC → Governance → Opportunities** tab (NORA Stage 6.6): governable improvement records classified by architecture domain (BA/AA/DA/TA) with priority and lifecycle (proposed → approved → in transition → realized / rejected), linked cards, and one-click assignment of the realising transition initiative (which automatically moves the opportunity to "in transition").
- **Architecture state on cards (current / transition / target).** Every card now carries an architecture-state slice plus a typed change marker (create / modify / replace / retire / consolidate) and an optional successor link — so a 3–5-year target blueprint lives side-by-side with the live landscape on the same inventory (NORA Stages 6–7, TOGAF baseline/target). A state badge on the card detail header switches slices; the inventory API filters by state; promoting a target card to current emits a dedicated audit event; state and change type travel in the workspace-transfer bundle.
- **Gap Analysis report.** New report under **Reports → Gap Analysis** bucketing every current→target delta (new / replacements / modifications / retirements), showing which transition initiative delivers each change (with its transition role), flagging **untraceable** changes no initiative owns, and offering one-click "Assign to initiative". Initiative relation types gain a `transitionRole` attribute (introduces / modifies / retires) when the NORA profile is applied.
- **Multi-step governance approval (stage gates).** Optional governance mode (**Admin → Settings → Governance Workflow**) replaces one-click card approval with an ordered review chain — by default Chief Architect, then EA Governance Committee. Cards are *submitted* for review (new IN_REVIEW status), each chain step is decided by users holding that role plus the new `governance.approve_step` permission, segregation of duties blocks submitters from approving their own round, mid-review edits restart the round, chain members get in-app notifications, and the card detail shows the live review-chain stepper. The classic flow is untouched when the mode is off.
- **NORA governance role pack.** Applying the NORA profile seeds three roles: **EA Working Team** (drafts, cannot approve), **Chief Architect** (authoring + first review step) and **EA Governance Committee** (read access + final approval).
- **NORA TRM metadata on Technology Standards.** The Technology Standards catalogue now carries the fields NORA's Technology Reference Model expects on a Service Standard: issuing body (DGA, NCA, W3C, …), mandate level (mandatory / recommended / optional), next review date, specification URL, TRM code, and an optional link to a **Tech Category** card so standards slot into the TRM Service Area / Category tree. All fields are editable in the standards dialog (with a Tech Category picker) and returned by the list / radar / detail endpoints; the Tech Category link participates in workspace export/import. Translated in all 10 locales.
- **Government Service card type (NORA Service Catalogue).** Applying the NORA profile now also creates a `GovService` card type in the Business Architecture layer — service code, beneficiary types (citizen / resident / business / government / visitor), delivery channels (portal / mobile app / service center / call center / kiosk), service maturity (informational → proactive), fee model, SLA and volume metrics — plus five relation types wiring services to Business Processes ("is realized by"), Applications ("is delivered by"), Organizations ("provides"), Business Capabilities ("supports") and Data Objects ("uses"), and a `service_owner` stakeholder role. Relation creation respects the one-relation-type-per-pair rule: pairs an admin already modelled are skipped, never duplicated. Fully translated in all 10 locales.
